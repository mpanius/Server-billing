# Server Billing Manager

Self-hosted панель для учета серверов, сроков оплаты и быстрых переходов к оплате у хостинг-провайдера.

Сервис не проводит платежи сам и не хранит банковские карты. Он напоминает о сроках, показывает нужный сервер, хранит ссылку на кабинет/оплату и помогает быстро перейти к провайдеру.

## Возможности

- Дашборд серверов, провайдеров и ближайших оплат.
- Поиск и фильтры по серверу, IP, аккаунту, провайдеру и статусу оплаты.
- Онбординг-чеклист после установки.
- Аккаунты хостинга: провайдер, логин, зашифрованный секрет, ссылка на кабинет, ссылка на оплату.
- Привязка нескольких серверов к одному аккаунту хостинга.
- Данные доступа к конкретному серверу: IP, страна/локация, логин и зашифрованный пароль.
- Шаблоны популярных провайдеров для быстрого заполнения ссылок, валюты и подсказок.
- **Каталог провайдеров** — компактный маркетплейс: ~36 хостеров, SVG-флаги ~57 стран, фильтры по цене, RAM, CPU, трафику и валюте, карточки с тарифами-ориентирами и badge «API docs» (ссылка на документацию, не live-sync).
- Обновление каталога и цен **без пересборки Docker**: remote JSON-bundle, кнопка «Обновить каталог» на странице **Провайдеры**, автообновление раз в сутки через scheduler.
- При добавлении сервера — поле «Шаблон провайдера» с поиском по буквам (без отдельного модального каталога).
- Статусы оплаты: в порядке, скоро, срочно, просрочено.
- Ввод даты следующей оплаты через календарь в форме сервера.
- Страница оплаты конкретного сервера.
- Кнопка перехода на ссылку оплаты сервера или аккаунта хостинга.
- Кнопка `Отметить оплачено`, которая переносит дату на следующий период.
- История оплат с датами, суммами и комментариями.
- Аналитика расходов по месяцам и провайдерам.
- Плановая месячная нагрузка по текущим серверам.
- Календарь оплат по месяцам с экспортом в iCal (`.ics`).
- Интеграция с BILLmanager-совместимыми провайдерами: тип интеграции у аккаунта, проверка подключения и синхронизация услуг (только чтение) — даты, суммы, статусы.
- Интеграция **Web API** провайдеров: синхронизация заказов, IP и дат окончания по API-ключу.
- Автосинхронизация услуг по расписанию с уведомлением об изменениях в Telegram.
- Защита ручных правок сервера от перезаписи синхронизацией (`Не перезаписывать при автосинхронизации`).
- SSH-порт у сервера (по умолчанию 22).
- Веб-терминал: SSH-подключение к серверу прямо из браузера через панель (xterm.js + WebSocket); выключен по умолчанию.
- Валюты `RUB`, `USD`, `EUR`, `USDT`.
- Курсы валют к RUB: фиат через ЦБ РФ, USDT через CoinGecko.
- Telegram-уведомления через отдельный reminder worker.
- Мастер Telegram в настройках: bot token → ссылка на бота → «Найти мой чат» → тест (chat id вручную не обязателен).
- Настройки Telegram, расписания, backup и каталога провайдеров в веб-панели.
- Мониторинг SSL-сертификатов доменов серверов и отдельного списка; уведомления в Telegram при истечении.
- Backup SQLite-базы в Telegram по расписанию или вручную.
- Ручной запуск проверки уведомлений из панели.
- Обновление сервиса из репозитория кнопкой в веб-панели.
- Ненавязчивая ссылка `Поддержать разработку` через Telegram Stars.
- Авторизация администратора по логину и паролю.
- **Белый список IP** — ограничение доступа к панели только с доверенных адресов и подсетей (CIDR).
- Смена пароля администратора в веб-панели.
- Страница подключения собственного домена (в разделе `Настройки`).
- Развертывание через Docker Compose и Caddy **или** через nginx на уже занятом 443 порту.
- HTTPS без домена через `IP.sslip.io` или собственный домен с сертификатом Caddy.

## Установка одной командой

На чистом Linux-сервере под `root` выполните:

```bash
curl -fsSL https://raw.githubusercontent.com/AlekseyRusaleev/Server-billing/main/scripts/install.sh | bash
```

Если вход выполнен не под `root`, используйте:

```bash
curl -fsSL https://raw.githubusercontent.com/AlekseyRusaleev/Server-billing/main/scripts/install.sh | sudo bash
```

Команда рассчитана на запуск из обычной SSH-консоли личного сервера: установщик интерактивно спрашивает домен, логин и пароль администратора.

Установщик спросит:

- домен, если хотите использовать свой HTTPS-домен;
- email для сертификатов Caddy, если указан домен;
- логин администратора;
- пароль администратора.

Также установщик попробует автоматически получить курсы валют: фиатные валюты с официального XML endpoint ЦБ РФ, `USDT` через CoinGecko. Если источники недоступны, сервис стартует с `RUB:1`, а курсы можно обновить позже в разделе `Настройки`.

Если домен не указан, установщик автоматически создаст HTTPS-адрес через `sslip.io`:

```text
https://YOUR_SERVER_IP.sslip.io
```

Обычный адрес `http://YOUR_SERVER_IP` будет редиректить на защищенную HTTPS-ссылку.

Если автоматический установщик не подходит, используйте [ручную установку](#ручная-установка-без-installsh).

## HTTPS без домена

Свой домен не обязателен. По умолчанию используется публичный DNS-алиас `sslip.io`, который указывает на IP сервера:

```text
192.0.2.10.sslip.io -> 192.0.2.10
```

Caddy автоматически выпускает сертификат Let's Encrypt для этого имени.

## Установка с доменом

1. Создайте DNS `A`-запись домена на IP сервера.
2. Запустите install script.
3. Введите домен, например:

```text
billing.example.com
```

Caddy автоматически получит и обновит HTTPS-сертификат.

## Настройки в панели

После входа откройте раздел `Настройки`.

**Мастер Telegram**

1. Вставьте bot token от `@BotFather` → «Проверить и сохранить».
2. Откройте бота по ссылке в мастере, нажмите **Start** (или напишите любое сообщение).
3. Нажмите **«Найти мой чат»** — панель подберёт чат автоматически; если их несколько, покажет список для выбора.
4. **«Отправить тест»** — проверка доставки уведомлений.

**Расписание и валюты**

- `Base URL` для ссылок в Telegram;
- дни напоминаний, например `7,3,1,0,-1`;
- периодичность проверки в секундах;
- периодичность backup базы в Telegram;
- базовая валюта аналитики: `RUB`, `USD`, `EUR` или `USDT`;
- курсы валют к RUB;
- ручной backup, проверка уведомлений, обновление курсов.

**Доступ к панели**

- белый список IP: разрешённые адреса и подсети (по одному на строку), кнопка «Добавить мой IP»;
- пустой список — доступ без ограничений;
- при сохранении непустого списка ваш текущий IP должен входить в него (защита от самоблокировки).

**Сервис**

- обновление приложения из репозитория (кнопка «Обновить сервис»);
- синхронизация каталога провайдеров (дублирует кнопку на странице **Провайдеры**);
- веб-терминал, смена пароля администратора.

**Домен и SSL**

- подключение своего домена и `Base URL`;
- мониторинг SSL-сертификатов, порог уведомлений, добавление доменов в список.

По умолчанию scheduler проверяет оплаты раз в сутки:

```env
CHECK_INTERVAL_SECONDS=86400
REMINDER_DAYS=7,3,1,0,-1
```

Значение `-1` означает напоминание о просроченной оплате.

Если `Backup в Telegram, дней` равен `0`, автоматическая отправка backup отключена.

Курсы фиатных валют берутся из ЦБ РФ:

```text
https://www.cbr.ru/scripts/XML_daily.asp
```

Курс `USDT/RUB` берется из CoinGecko Simple Price API. Если CoinGecko временно недоступен, сервис может временно использовать `USD/RUB` как запасной курс для `USDT`.

Сервис хранит курсы в формате:

```text
RUB:1,USD:90.1234,USDT:89.9000,EUR:98.1234
```

Плановая месячная нагрузка считается по текущим серверам:

```text
amount / billing_period_days * 30
```

## Каталог провайдеров

Раздел **Провайдеры** — личный справочник хостеров, не полноценный биллинг. Компактные карточки, переход на сайт провайдера или по `referral_url`, тарифы как **ориентиры** с датой актуальности (например, «11 июня 2026»).

### Что на странице

- **Панель каталога** — дата цен, источник (`online` / `из образа`), кнопка **«Обновить каталог»**.
- **Флаги стран** — сетка из ~57 локаций, фильтр по стране.
- **Фильтры** — цена от/до, валюта, RAM, CPU, трафик, «API docs».
- **Карточки** — min/max цена, флаги локаций, переход в детали с тарифами.
- **Промо** — партнёрские блоки из `app/provider_catalog.json`.

При добавлении сервера тот же каталог доступен в поле **«Шаблон провайдера»** с поиском по буквам. Кнопка «Создать сервер» на странице провайдера открывает форму с уже выбранным шаблоном.

Badge **«API docs»** — ссылка на документацию. Badge **«BILLmanager»** / **«API sync»** — для аккаунта доступна живая синхронизация (см. [Интеграция с провайдерами](#интеграция-с-провайдерами)).

### Добавление провайдера (API + реф-ссылка)

Присылайте **документацию API** и **реферальную ссылку** — для каждого хостера добавляется:

1. `app/provider_templates.json` — `referral_url`, `api_docs_url`, `integration_type`, ссылки кабинета;
2. `app/provider_plans.json`, `app/provider_locations.json` — тарифы и страны;
3. коннектор и тип в `app/integrations.py` — если API позволяет read-only sync.

Шаблон полей: `scripts/provider_entry.example.json`. Провайдеры с готовым коннектором отмечены в каталоге badge **«API sync»**.

### Файлы каталога

```text
app/provider_templates.json      — список провайдеров
app/provider_plans.json          — тарифы-ориентиры, prices_as_of
app/provider_locations.json      — страны по доменам
app/provider_countries.json      — названия стран
app/provider_catalog.json        — notice, promos
app/static/flags/*.svg           — флаги для UI
app/provider_bundle.example.json — формат remote-bundle
```

### Обновление цен без пересборки

Каталог можно обновлять **без `git pull` и без пересборки Docker**:

1. Опубликуйте JSON-bundle по HTTPS (формат — `app/provider_bundle.example.json`): GitHub Raw, CDN или свой сервер.
2. В `.env` задайте:

```env
PROVIDER_CATALOG_URL=https://example.com/provider_bundle.json
```

3. **Scheduler** подтягивает bundle раз в сутки (`CHECK_INTERVAL_SECONDS`, по умолчанию 86400).
4. Кнопка **«Обновить каталог»** на странице **Провайдеры** — ручное обновление за несколько секунд с перезагрузкой страницы.

Bundle может обновлять: `plans_by_domain`, `countries_by_domain`, `countries`, `promos`, `providers`, `prices_as_of`.

Кэш на диске: `data/catalog_cache/provider_bundle.json`. Если `PROVIDER_CATALOG_URL` не задан, используется встроенный каталог из образа; кнопка «Обновить каталог» неактивна.

У шаблона может быть поле `referral_url`. Если оно заполнено, кнопка «На сайт провайдера» ведёт по партнёрской ссылке. Иначе — `website_url`.

Если нужного провайдера нет в списке, выберите **«Заполнить вручную»** и введите данные сами.

## Календарь оплат

Раздел `Календарь` показывает оплаты серверов по месяцам с навигацией и подсветкой сегодняшнего дня. Кнопка `Экспорт в календарь (iCal)` отдает файл `.ics`, который можно подписать в Google Calendar, Apple Calendar или Outlook. Для каждой оплаты создается событие на дату следующего платежа с напоминанием за день.

## Интеграция с провайдерами

У аккаунта хостинга можно выбрать тип интеграции:

- **Ручной** — данные вводятся вручную (по умолчанию);
- **BILLmanager** — BILLmanager-совместимый API (URL биллинга, логин, пароль);
- **Web API** — REST/Web API провайдера (API-ключ в поле секрета).

Для интеграций доступны кнопки **Проверить** и **Синхронизировать** в карточке аккаунта. Синхронизация только на чтение: услуги, IP, даты окончания, статусы. Платежи через API не выполняются.

### BILLmanager

Укажите URL биллинга (например `https://my.example.com/billmgr`), логин и пароль от кабинета. Подтягиваются VPS/услуги, IP, суммы и даты продления. Шаблон провайдера в каталоге подставляет URL и тип интеграции автоматически.

### Web API

Для провайдеров с собственным API выберите шаблон в каталоге (badge **«API sync»**), вставьте ключ в поле секрета, включите **Автосинхронизацию** → **Проверить** → **Синхронизировать**.

Чтобы синхронизация не затирала ручные правки конкретного сервера, включите у него флаг `Не перезаписывать при автосинхронизации`. Scheduler обновляет аккаунты с автосинком по расписанию и присылает сводку изменений в канал уведомлений.

## Веб-терминал

Веб-терминал позволяет открыть SSH-сессию к серверу прямо из браузера. На карточке сервера и на странице его редактирования есть кнопка `Терминал`.

Как это работает:

- SSH-подключение идет с сервера панели к целевому серверу, а не с вашего компьютера.
- В браузер по WebSocket летит только интерактивная сессия терминала (xterm.js).
- Пароль расшифровывается только на сервере панели и в браузер не передается.
- Кнопка неактивна, если у сервера не заполнены IP, логин и пароль.
- У сервера можно задать `SSH порт` (по умолчанию 22).

Включение и безопасность:

- Функция выключена по умолчанию. Включите ее в `Настройки -> Веб-терминал`.
- Доступ к терминалу требует входа в панель; WebSocket дополнительно проверяет сессию и белый список IP.
- Ключ хоста проверяется по схеме TOFU (trust on first use). При первом подключении ключ запоминается в `data/ssh_known_hosts.json`, а при его смене подключение разрывается с предупреждением о возможной MITM-атаке. Если вы переустановили сервер, удалите старую запись из этого файла.
- Неактивная сессия автоматически закрывается через 15 минут.
- Команды не логируются.

Важный сетевой нюанс: целевой сервер должен принимать SSH-подключения с IP сервера панели. Если firewall провайдера разрешает SSH только с вашего домашнего IP, подключение из панели не пройдет.

## Production Docker Compose

Production-режим использует:

- `app` — FastAPI web app;
- `scheduler` — Telegram reminder worker и проверка SSL;
- `caddy` — reverse proxy и HTTPS (по умолчанию);
- `updater` — внутренний сервис для обновления из веб-панели.

Стандартный запуск после настройки `.env`:

```bash
docker compose -f docker-compose.prod.yml up -d --build
```

## Ручная установка (без install.sh)

Автоматический `scripts/install.sh` удобен на чистом VPS, но не всем подходит. Ручная установка нужна, если:

- хотите свой каталог, не `/opt/server-billing`;
- на сервере уже заняты 80/443 другим веб-сервером (nginx, Apache, другой сайт);
- нет доступа под `root` или нельзя запускать curl|bash;
- нужно заранее посмотреть и поправить каждый шаг.

### 1. Зависимости

```bash
apt update
apt install -y git python3
# Docker Engine + Compose plugin — по официальной инструкции:
# https://docs.docker.com/engine/install/
```

Проверка:

```bash
docker compose version
git --version
python3 --version
```

### 2. Клонирование

```bash
git clone https://github.com/AlekseyRusaleev/Server-billing.git /opt/server-billing
cd /opt/server-billing
```

Каталог можно заменить на свой, главное — потом указать его же в `INSTALL_DIR` для updater (по умолчанию `/opt/server-billing`).

### 3. Файл `.env`

```bash
cp .env.example .env
```

Сгенерируйте секреты и хеш пароля администратора:

```bash
python3 - <<'PY'
import base64, hashlib, os, secrets

password = input("Пароль администратора: ").encode()
salt = os.urandom(16)
digest = hashlib.pbkdf2_hmac("sha256", password, salt, 260_000)
print("APP_SECRET_KEY=" + secrets.token_urlsafe(48))
print("APP_ENCRYPTION_KEY=" + base64.urlsafe_b64encode(os.urandom(32)).decode())
print("APP_UPDATE_TOKEN=" + secrets.token_urlsafe(32))
print(
    "ADMIN_PASSWORD_HASH=pbkdf2_sha256:260000:"
    + base64.urlsafe_b64encode(salt).decode()
    + ":"
    + base64.urlsafe_b64encode(digest).decode()
)
PY
```

Скопируйте вывод в `.env` и допишите минимум:

```env
ADMIN_USERNAME=admin
SERVER_IP=YOUR_SERVER_IP
BASE_URL=https://YOUR_SERVER_IP.sslip.io
CADDY_SITE_ADDRESS=YOUR_SERVER_IP.sslip.io
APP_UPDATE_URL=http://updater:8765/update
```

Для своего домена:

```env
BASE_URL=https://billing.example.com
CADDY_SITE_ADDRESS=billing.example.com
CADDY_EMAIL=admin@example.com
```

`TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID` можно оставить пустыми — настроите в панели после входа.

Полный список переменных — в разделе [Настройки `.env`](#настройки-env).

### 4. Запуск контейнеров

**Обычный случай** — порты 80 и 443 свободны, HTTPS через встроенный Caddy:

```bash
docker compose -f docker-compose.prod.yml up -d --build
```

Проверка:

```bash
docker compose -f docker-compose.prod.yml ps
curl -I "https://$(grep '^CADDY_SITE_ADDRESS=' .env | cut -d= -f2-)/login"
```

**Если 80/443 уже заняты** — см. [Свой reverse proxy](#свой-reverse-proxy-nginx-apache-и-др) ниже; Caddy из compose в этом случае не запускайте.

### 5. Первый вход

1. Откройте `BASE_URL` из `.env`.
2. Войдите логином и паролем из шага 3.
3. В `Настройки` пройдите мастер Telegram: token → откройте бота → «Найти мой чат» → тест.
4. Добавьте первый сервер или удалите демо-данные.

### 6. Обновление вручную

Перед обновлением сохраните папку `data/` (там SQLite-база и ключи SSH для терминала).

**Стандартная установка (Caddy):**

```bash
cd /opt/server-billing
git pull --ff-only
docker compose -f docker-compose.prod.yml up -d --build
```

**Без Caddy** (свой reverse proxy):

```bash
cd /opt/server-billing
git pull --ff-only
docker compose -f docker-compose.nginx.yml up -d --build
```

Кнопка «Обновить сервис» в панели делает то же самое автоматически, если в `.env` заданы `APP_UPDATE_URL` и `APP_UPDATE_TOKEN`.

## Свой reverse proxy (nginx, Apache и др.)

Если на сервере уже работает nginx, Apache или другой прокси с HTTPS — **не поднимайте Caddy** из `docker-compose.prod.yml`, иначе будет конфликт за 80/443.

Схема:

```text
Интернет → ваш nginx :443
              ↓ proxy_pass
         127.0.0.1:8000 → Docker (app + scheduler + updater)
```

Запуск без Caddy:

```bash
docker compose -f docker-compose.nginx.yml up -d --build
```

Приложение слушает только `127.0.0.1:8000`. Настройте прокси на хосте с поддержкой **WebSocket** (нужен для веб-терминала). Готовый пример для nginx: [`deploy/nginx/server-billing.conf.example`](deploy/nginx/server-billing.conf.example).

В `.env` укажите публичный `BASE_URL=https://ваш-домен` — без `CADDY_SITE_ADDRESS`, он для Caddy не нужен.

Updater в `docker-compose.nginx.yml` уже настроен пересобирать только `app` и `scheduler`, без Caddy.

## Настройки `.env`

```env
APP_NAME=Server Billing Manager
DATABASE_PATH=/app/data/server_billing.db
BASE_URL=https://YOUR_SERVER_IP.sslip.io
SERVER_IP=YOUR_SERVER_IP
CADDY_SITE_ADDRESS=YOUR_SERVER_IP.sslip.io
CADDY_EMAIL=
APP_SECRET_KEY=
APP_ENCRYPTION_KEY=
ADMIN_USERNAME=admin
ADMIN_PASSWORD_HASH=
TELEGRAM_BOT_TOKEN=
TELEGRAM_CHAT_ID=
REMINDER_DAYS=7,3,1,0,-1
CHECK_INTERVAL_SECONDS=86400
BACKUP_INTERVAL_DAYS=7
CURRENCY_BASE=RUB
CURRENCY_RATES=RUB:1
CURRENCY_RATES_UPDATED_AT=
APP_UPDATE_URL=http://updater:8765/update
APP_UPDATE_TOKEN=
PROVIDER_CATALOG_URL=
PANEL_IP_ALLOWLIST=
TRUSTED_PROXIES=
```

`PROVIDER_CATALOG_URL` — HTTPS-ссылка на JSON-bundle каталога провайдеров. Пустое значение = только встроенный каталог из образа.

`PANEL_IP_ALLOWLIST` — начальный белый список IP для доступа к панели (через запятую или с новой строки). Пустое значение = без ограничений. После сохранения в веб-панели приоритет у настройки из SQLite.

Пример:

```env
PANEL_IP_ALLOWLIST=203.0.113.10,192.168.1.0/24
```

`TRUSTED_PROXIES` — дополнительные доверенные прокси (IP или CIDR), с которых принимаются `X-Forwarded-For` / `X-Real-IP`. По умолчанию доверяются только `127.0.0.0/8` и `::1` (локальный reverse proxy). Для Docker-сети укажите IP контейнера прокси, например `172.18.0.2/32`. Прямые запросы с чужого IP с подделанными заголовками не обходят whitelist.

Для домена:

```env
BASE_URL=https://billing.example.com
CADDY_SITE_ADDRESS=billing.example.com
CADDY_EMAIL=admin@example.com
```

## Telegram

Telegram настраивается после установки в веб-панели (`Настройки → Мастер настройки`). Установщик не спрашивает token и chat id в терминале.

**Подключение за 4 шага**

1. Создайте бота через `@BotFather`, скопируйте token.
2. В панели: **«Проверить и сохранить»** — token проверяется через Telegram API и хранится только на вашем сервере (SQLite / `.env`).
3. Откройте бота по ссылке в мастере → **Start** (или любое сообщение) → **«Найти мой chat»**. Панель снимает webhook (если мешал), читает входящие сообщения и предлагает выбрать чат из списка.
4. **«Отправить тест»** — убедитесь, что уведомления приходят.

Chat id вручную не обязателен. Если автопоиск не сработал — раскройте «Вставить chat id вручную» в мастере или используйте @userinfobot / @getmyid_bot.

Если `TELEGRAM_BOT_TOKEN` или chat не настроены, scheduler работает, но уведомления, backup и SSL-алерты в Telegram не отправляются.

## Безопасность

Панель рассчитана на **личный self-hosted** сервер за HTTPS reverse proxy. Ниже — что уже встроено в код и деплой, и что нужно настроить вручную.

### Обязательные секреты

Без этих переменных в `.env` панель **не пускает** (fail-closed):

| Переменная | Назначение |
|------------|------------|
| `APP_SECRET_KEY` | Подпись сессионных cookie |
| `ADMIN_PASSWORD_HASH` | Пароль администратора (PBKDF2-SHA256) |
| `APP_ENCRYPTION_KEY` | Шифрование паролей серверов, API-ключей и token бота уведомлений в БД |

Новые секреты **не сохраняются**, пока не задан `APP_ENCRYPTION_KEY`. Не меняйте этот ключ после начала работы — старые записи не расшифруются.

### Аутентификация и сессии

- Одна учётная запись администратора (`ADMIN_USERNAME`, по умолчанию `admin`).
- Сессия — подписанная cookie (`HttpOnly`, `SameSite=Lax`, `Secure` за HTTPS), срок 30 дней.
- При смене пароля в настройках **все старые сессии аннулируются**; текущая перевыпускается автоматически.
- На `/login` — rate limit: 5 неудачных попыток с одного IP за 15 минут.
- Все POST-формы защищены **CSRF-токеном** (double-submit cookie).

### Сеть и периметр

- **Prod:** приложение не торчит наружу — только reverse proxy на 80/443 (`docker-compose.prod.yml`).
- **Dev:** `docker-compose.yml` биндит `127.0.0.1:8000`, не `0.0.0.0`.
- **Белый список IP** (`Настройки → Доступ к панели` или `PANEL_IP_ALLOWLIST`): отдельные адреса и CIDR. Пустой список = без ограничений. Для SSH-туннеля с сервера добавьте `127.0.0.1` **явно** — автоматического bypass для localhost нет.
- За reverse proxy IP клиента берётся из `X-Forwarded-For` / `X-Real-IP` **только от доверенного прокси** (по умолчанию `127.0.0.0/8`, `::1`). В prod-compose для прокси в той же контейнерной сети задан `TRUSTED_PROXIES=172.16.0.0/12`; если прокси на хосте — обычно достаточно дефолта. Дополнительные адреса — `TRUSTED_PROXIES` в `.env`.
- Заголовки: HSTS, `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy` (на прокси, в примере конфига и в самом приложении).

### Хранение данных

- SQLite в `./data/server_billing.db`; контейнер приложения работает от **непривилегированного пользователя** (`uid 1000`). Каталог `data/` на хосте: `sudo chown -R 1000:1000 data`.
- Пароли SSH, API-ключи провайдеров, token бота — **шифрование в БД**.
- На странице оплаты пароли и секреты **не выводятся в HTML** — только «задан» / «не задан»; полные значения — в правке сервера/аккаунта.

### Исходящие запросы (anti-SSRF)

- URL в формах — только `http://` / `https://`; схемы вроде `javascript:` отклоняются.
- Коннекторы провайдеров: whitelist доменов по типу интеграции, блок private/link-local/metadata IP.
- BILLmanager — только **HTTPS**, учётные данные только в **POST**-теле (не в query string).
- SSL-монитор подключается только к публичным хостам.

### Резервное копирование

- Ручной и автоматический backup отправляет **зашифрованный** `.db.enc`, не сырой SQLite.
- При утечке token бота смените его; доступ к чату backup = доступ к зашифрованному файлу (ключ — `APP_ENCRYPTION_KEY` на сервере).

### Веб-терминал

- Выключен по умолчанию (`Настройки → Веб-терминал`).
- WebSocket не принимается без авторизации и проверки IP whitelist.
- SSH с сервера панели; пароль не уходит в браузер.
- Проверка ключа хоста (TOFU) **до** отправки пароля; при смене ключа — разрыв с предупреждением.
- Сессия закрывается после 15 мин бездействия; команды не логируются.

### Обновление из панели

- Сервис `updater` в prod-compose: `git pull` + `docker compose up --build` по токену.
- `APP_UPDATE_TOKEN` — **минимум 32 символа**; `APP_UPDATE_URL` — только публичный **HTTPS**.
- Updater не публикует порт наружу, но имеет доступ к сокету контейнерной платформы — храните токен как root-пароль сервера.

### Чеклист prod

```
□ APP_SECRET_KEY, ADMIN_PASSWORD_HASH, APP_ENCRYPTION_KEY заданы
□ Панель только через HTTPS (reverse proxy), не docker-compose.yml с :8000 на 0.0.0.0
□ PANEL_IP_ALLOWLIST настроен осознанно (при необходимости — 127.0.0.1 для туннеля)
□ TRUSTED_PROXIES = только реальный reverse proxy
□ data/ принадлежит uid 1000 (после обновления образа)
□ Веб-терминал выключен, если не нужен
□ APP_UPDATE_TOKEN длинный (≥32) и не светится
□ Демо-данные удалены после первого запуска
```
